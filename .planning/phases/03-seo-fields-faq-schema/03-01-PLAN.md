<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin browser → Admin::BlogsController | Authenticated admin submits keywords via form; strong params filter |
| DB → public blog show page | Stored jsonb array surfaces in `<meta name="keywords">` content attribute |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Admin::BlogsController#update (keywords param) | mitigate | Inherits Admin::BaseController → `authenticate_user!` + `ensure_admin!`; unauthenticated/non-admin requests rejected before reaching action |
| T-03-02 | Tampering | blog_params strong params | mitigate | `keywords: []` permits only an array of scalars at top level; nested hashes/keys rejected by Rails strong params |
| T-03-03 | Tampering / Injection | `<meta name="keywords" content="<%= page_keywords %>">` | mitigate | ERB auto-escapes the content attribute; no `raw`/`html_safe`/`sanitize` on the keywords output path; values are HTML-attribute-encoded |
| T-03-04 | Information Disclosure | Keywords visible in public HTML | accept | Keywords are intentionally public SEO metadata; admin must not enter secrets there (documented in form hint copy) |
| T-03-05 | Tampering | Stimulus `renderChips` rendering user-typed strings into chip DOM | mitigate | Use `document.createTextNode` / `el.textContent = kw` for chip label — never `innerHTML` for the keyword value; data-attribute index uses parseInt |
| T-03-06 | Denial of Service | Unbounded keywords array length | accept | jsonb has no enforced cap; admin-only authoring with audit log; oversized arrays would degrade meta tag readability long before becoming a DB concern |
</threat_model>
