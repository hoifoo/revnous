| T-03-13 | Tampering | Non-image upload masquerading as og_image (e.g. .php, .html) | mitigate | Model-layer validation `validate_og_image_content_type` uses an explicit raster-image whitelist (see T-03-14) AND purges the attachment on failure; HTML `accept="image/*"` is UX-only and not relied on for security |
| T-03-14 | Tampering / Stored XSS via SVG | SVG upload with embedded `<script>` | mitigate | Applied to Task 1 action: `validate_og_image_content_type` requires content_type to be in `%w[image/png image/jpeg image/jpg image/gif image/webp]` (explicit whitelist), NOT a `start_with?("image/")` prefix check — which would accept `image/svg+xml`. | T-03-15 | Denial of Service | Very large image upload | accept | ActiveStorage default service config governs size; admin-only authoring; existing cover photo flow has same exposure surface and has not been an issue |
| T-03-16 | Information Disclosure | OG image URL public | accept | OG image is intentionally public; admin must not upload private/internal images |
