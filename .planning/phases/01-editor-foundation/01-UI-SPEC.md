# Phase 1 — UI Design Contract: Editor Foundation

> Visual and interaction contract for Phase 1. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Scope: Tiptap editor replacement on the admin blog form, sticky toolbar, WYSIWYG heading styles, content migration, JSON-LD security fix.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — plain Tailwind CSS 4 (Rails/ERB; shadcn not applicable) |
| Preset | not applicable |
| Component library | none |
| Icon library | inline SVG only — match Trix icon style already present in actiontext.css |

## Spacing Scale

Tailwind CSS 4 default 4px base scale applies throughout. Specific values in use:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps within toolbar button groups, separator gaps |
| sm | 8px | Button internal padding (compact toolbar buttons) |
| md | 16px | Default element spacing; form field gaps; toolbar horizontal padding |
| lg | 24px | Section padding within form card; spacing between toolbar and editor |
| xl | 32px | Form card internal padding (matches existing `p-8` = 32px) |
| 2xl | 48px | Major section separation (matches existing `space-y-6` stacks) |
| 3xl | 64px | Admin navbar height (`h-16` = 64px); sticky toolbar `top` offset |

Exceptions:
- Toolbar `top` sticky offset: `top-16` (64px) to clear the `h-16` admin navbar exactly. If a CSS variable is preferred for maintainability, define `--admin-navbar-height: 4rem` in `application.tailwind.css` and use `top-[var(--admin-navbar-height)]`.
- Toolbar button minimum touch target: 36px height (`h-9`) — below the 44px mobile target because this is a desktop admin tool. Acceptable for the admin-only audience.
- Disabled stub button size: identical to active buttons — no visual size difference, only opacity and cursor change.

## Typography

Admin UI uses Tailwind's default type scale (no custom font). Two weights and four size roles apply to Phase 1 output:

**Weights (exactly 2):** `font-normal` (400) for body text and `font-semibold` (600) for all labels, toolbar text, and headings.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Label | 14px (`text-sm`) | 600 (`font-semibold`) | 1.25 | Form field labels (`block text-sm font-semibold text-gray-700`) |
| Body | 16px (`text-base`) | 400 (`font-normal`) | 1.5 | Helper text, editor body default, hint paragraphs |
| Heading | 30px (`text-3xl`) | 600 (`font-semibold`) | 1.2 | Page headings ("Edit Blog Post", "New Blog Post") |
| Toolbar label | 12px (`text-xs`) | 600 (`font-semibold`) | 1 | Heading-level dropdown label ("H1"–"H6") rendered as text buttons |

**WYSIWYG heading sizes inside `.ProseMirror`:** Controlled entirely by `prose prose-lg` (Tailwind Typography). Do not declare custom heading sizes — they must match the public blog show page rendering automatically. The executor must not override `prose` heading sizes with custom CSS.

**Source:** Detected from `app/views/admin/blogs/edit.html.erb`, `new.html.erb`, `_form.html.erb`.

## Color

All colors are Tailwind CSS 4 palette values (no custom hex). Derived from existing admin views.

| Role | Tailwind Value | Hex (reference) | Usage |
|------|---------------|-----------------|-------|
| Dominant (60%) | `gray-100` / `gray-50` | #f3f4f6 / #f9fafb | Page background (`bg-gray-100`), form card backgrounds |
| Secondary (30%) | `white` / `gray-200` | #ffffff / #e5e7eb | Form card surface (`bg-white`), borders, dividers, toolbar background |
| Accent (10%) | `pink-600` | #db2777 | Submit button, focus rings, active toolbar button indicator, checkbox checked state |
| Destructive | `red-500` / `red-700` | #ef4444 / #b91c1c | Validation error border (`border-red-500`), error text (`text-red-700`), error background (`bg-red-50`) |

Accent reserved for:
1. The "Save blog post" / "Update Blog Post" submit button (`bg-pink-600 hover:bg-pink-700`)
2. Input focus rings (`focus:ring-2 focus:ring-pink-500`)
3. Active toolbar button state — button background tint when format is active (e.g. Bold is ON): `bg-pink-50 text-pink-700`
4. Checkbox checked indicator (`text-pink-600`)

Accent is NOT used for:
- Toolbar button default state (use `text-gray-700` on `bg-white`)
- Disabled stub buttons (use `text-gray-400 opacity-50`)
- General text or labels

**Toolbar active state detail:** When a Tiptap mark or node is active (e.g. cursor is inside bold text), the corresponding toolbar button shows: `bg-pink-50 text-pink-700 rounded`. When inactive: `text-gray-700 hover:bg-gray-100 rounded`.

**Disabled stub state (table, image buttons):** `text-gray-400 opacity-50 cursor-not-allowed`. No hover effect. No click handler.

## Component Specifications

**Primary focal point:** The `bg-pink-600` submit button draws the eye as the only accent-colored element in the form. All other interactive elements use gray-scale so the CTA stands out without competition.

### Tiptap Editor Container

The editor replaces `form.rich_text_area :content` in `app/views/admin/blogs/_form.html.erb`.

```
<div class="tiptap-editor" data-controller="tiptap-editor">
  <!-- Toolbar (sticky) -->
  <div class="tiptap-toolbar sticky top-16 z-10 bg-white border border-gray-300 rounded-t-md px-3 py-2 flex items-center gap-1 flex-wrap">
    <!-- toolbar buttons -->
  </div>
  <!-- Editor area -->
  <div class="ProseMirror prose prose-lg max-w-none min-h-[400px] px-4 py-3 border border-t-0 border-gray-300 rounded-b-md focus:outline-none focus:ring-2 focus:ring-pink-500"
       data-tiptap-editor-target="editor"
       contenteditable="true">
  </div>
  <!-- Hidden input for form submission -->
  <input type="hidden" name="blog[body]" data-tiptap-editor-target="input">
</div>
```

**Sticky toolbar behavior:**
- `position: sticky`, `top: 64px` (= `top-16` in Tailwind), `z-index: 10`
- Background: `bg-white` (not transparent) so content scrolls behind it without showing through
- Bottom border only between toolbar and editor, not around the outside — achieved by `rounded-t-md` on toolbar and `rounded-b-md border-t-0` on editor area

**Editor minimum height:** `min-h-[400px]` (400px). Grows with content (no max-height, no scroll inside editor).

**Scope for `prose` classes:** Apply as `class="ProseMirror prose prose-lg max-w-none ..."` directly on the contenteditable div. If admin layout styles bleed into `prose` headings, add a wrapping selector `.tiptap-editor .ProseMirror` in `application.tailwind.css`:
```css
.tiptap-editor .ProseMirror {
  @apply prose prose-lg max-w-none;
}
```
This is the only permitted custom CSS addition for Phase 1. Do not duplicate heading font sizes.

### Toolbar Buttons

Each active toolbar button:
- Size: `h-9 w-9` (36x36px) for icon-only buttons; `h-9 px-2 text-xs font-semibold` for text-label buttons (H1–H6)
- Default: `inline-flex items-center justify-center rounded text-gray-700 hover:bg-gray-100 transition-colors`
- Active: add `bg-pink-50 text-pink-700`
- Icon: inline SVG, 16x16px (`w-4 h-4`), `aria-hidden="true"`
- Each button: `title="Bold"` (native tooltip) for accessibility

Toolbar button groups (separated by a `|` divider — `<div class="w-px h-6 bg-gray-200 mx-1"></div>`):
1. **Heading group:** H1, H2, H3, H4, H5, H6 — text-label buttons
2. **Inline format group:** Bold, Italic, Strikethrough — icon buttons
3. **List group:** Bullet list, Ordered list — icon buttons
4. **Link group:** Link — icon button
5. **History group:** Undo, Redo — icon buttons
6. **Stub group (disabled):** Table insert, Image upload — icon buttons with `opacity-50 cursor-not-allowed`

### Disabled Stub Buttons

```
<button type="button"
        class="inline-flex items-center justify-center h-9 w-9 rounded text-gray-400 opacity-50 cursor-not-allowed"
        disabled
        aria-disabled="true"
        title="Coming soon">
  <!-- icon SVG -->
</button>
```

Tooltip text: "Coming soon" — delivered via native `title` attribute. No external tooltip library.

### "Coming soon" tooltip implementation

Use `title="Coming soon"` on the `<button>` element. Native browser tooltip appears on hover. No JavaScript required. No third-party tooltip library.

### Form Field: Body (replacing Trix field)

The existing form label stays:
```erb
<%= form.label :body, "Content", class: "block text-sm font-semibold text-gray-700 mb-2" %>
```
Label text: "Content" (was using `:content`; the new field is `:body`). Helper text below: "Supports rich text and heading formatting" (replaces old "Supports rich text, images, and formatting" — images are Phase 2).

### Error Display (validation failures)

Existing error block pattern reused without modification:
```erb
<div class="bg-red-50 border-l-4 border-red-500 p-4">
  <h3 class="text-sm font-semibold text-red-800 mb-2">N errors prohibited this blog from being saved:</h3>
  <ul class="list-disc list-inside text-sm text-red-700">...</ul>
</div>
```

## Interaction Contracts

### Toolbar sticky scroll
- Toolbar remains pinned at `top: 64px` while user scrolls a long post
- Admin navbar (64px) sits above; toolbar does not overlap nav
- Content scrolls behind toolbar without clipping — `z-10` on toolbar, content at default z-index

### Toolbar button active state update
- On every Tiptap `selectionUpdate` and `transaction` event, the Stimulus controller reads `editor.isActive('bold')`, `editor.isActive('heading', { level: 1 })`, etc.
- Active buttons receive `bg-pink-50 text-pink-700`; inactive buttons revert to `text-gray-700`
- State update is synchronous (no debounce needed)

### Form submission
- On form submit, Stimulus controller writes `editor.getHTML()` into the hidden `blog[body]` input
- Rails controller receives `:body` as a permitted param
- Sanitization runs in `before_save` on the Blog model — not in the controller

### Link insertion
- Clicking the Link toolbar button: opens a native `window.prompt("Enter URL:")` dialog
- If URL is entered: `editor.chain().focus().setLink({ href: url }).run()`
- If prompt is cancelled or empty: no action
- No custom modal or overlay — native browser prompt is sufficient for admin tool

### Migration rake task output
- Visual contract: no UI element — this is a CLI operation
- Progress output format: `Migrated {n}/{total} posts` printed to stdout
- Rake task is idempotent (safe to re-run)

## States Required

Every interactive element must handle all of these states:

| Element | States |
|---------|--------|
| Toolbar button (active) | default, hover, active (format on), focus-visible |
| Toolbar button (disabled stub) | disabled only — no hover, no focus |
| Editor area | default, focused (ring-2 ring-pink-500), scrolled-content |
| Form submit button | default, hover, focus-visible, loading (form submitting) |
| Link button | default, hover, active (link at cursor) |
| Error banner | visible (N > 0 errors) / hidden (no errors) |

## Accessibility Contract

- All toolbar buttons: `type="button"` (prevent accidental form submit)
- All toolbar buttons: `title` attribute with plain-English action name
- Disabled stub buttons: `disabled` attribute + `aria-disabled="true"`
- Editor div: `role="textbox"` `aria-multiline="true"` `aria-label="Blog content"`
- Active format buttons: `aria-pressed="true"` when format is active, `aria-pressed="false"` when inactive
- Error banner: `role="alert"` on the error container div
- Keyboard: Tab moves into toolbar; arrow keys navigate between toolbar buttons (roving tabindex); Tab exits toolbar into editor

## Security Contract (SEC-01 and SEC-02)

These are behavioral, not visual, but documented here so the executor has the complete picture:

**SEC-01 — HTML sanitization:**
- Sanitize via `Rails::Html::SafeListSanitizer` in `before_save :sanitize_body` on `Blog` model
- Safelist: `p, br, h1, h2, h3, h4, h5, h6, ul, ol, li, strong, em, a, blockquote, code, pre`
- Allowed attributes on `a`: `href`, `target`, `rel`
- View renders with `raw @blog.body` — sanitization is at save time, not render time

**SEC-02 — JSON-LD injection fix:**
- Replace `schema.to_json.html_safe` with `json_escape(schema.to_json)` in all three helpers:
  - `render_article_schema`
  - `render_breadcrumbs_schema`
  - `render_product_schema`
  - `render_organization_schema` (fix proactively — same bug pattern)
- The `content_tag :script` wrapper applies HTML escaping; `json_escape` prevents `</script>` injection inside the JSON payload
