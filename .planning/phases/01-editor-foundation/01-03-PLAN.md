Concrete identifiers and structure required:
- Top-level: `namespace :blogs do ... end`.
- Inside the namespace: `desc "Migrate blog content from ActionText to blogs.body column"` followed by `task migrate_body: :environment do ... end` (the `:environment` symbol is required so the Rails app is loaded).
- Inside the task body, in order:
  1. `total = Blog.count`
  2. `migrated = 0`
  3. `skipped = 0`
  4. `Blog.find_each do |blog| ... end` (use `find_each` not `all.each` for batching memory safety)
  5. Inside the loop, first the idempotent skip guard: `if blog.body.present?; puts "Skipping post #{blog.id} — body already populated"; skipped += 1; next; end`
  6. Then look up the ActionText row: `rich_text = ActionText::RichText.find_by(record_type: "Blog", record_id: blog.id, name: "content")`
  7. If `rich_text.nil? || rich_text.read_attribute(:body).blank?`: `puts "Skipping post #{blog.id} — no ActionText content found"; skipped += 1; next`
  8. Otherwise: `raw_html = rich_text.read_attribute(:body)` (use `read_attribute` per RESEARCH Open Question 3 — the `.body` accessor returns an `ActionText::Content` object that triggers attachment rendering via the view layer; `read_attribute(:body)` returns the raw stored string)
  9. Strip attachments via Nokogiri: `doc = Nokogiri::HTML.fragment(raw_html); doc.css("action-text-attachment").each(&:remove); clean_html = doc.to_html`
  10. Write the cleaned HTML: `blog.update_column(:body, clean_html)` (use `update_column` to bypass `before_save :sanitize_body` — the ActionText HTML is already from a controlled source and full safelist sanitization would happen redundantly + the rake task should not require model-level callbacks to be loaded)
  11. `migrated += 1`
  12. `puts "Migrated #{migrated}/#{total} posts"` (use the post-increment counter so output starts at 1, not 0; per UI-SPEC Copywriting Contract)
- After the loop: `puts "Done. #{migrated} posts migrated, #{skipped} skipped."` (matches the format in UI-SPEC Copywriting Contract).

Do NOT add transaction wrapping (per-record `update_column` is atomic; wrapping the whole loop in a transaction would hold a long lock during a production deploy — re-runnability is the safety mechanism per D-01).

Do NOT delete or alter rows in `action_text_rich_texts` — they remain as the rollback path per D-03.

Do NOT add a `:reverse` / `rollback` task. If a rollback is needed, the operator can run `Blog.update_all(body: nil)` manually and re-cut over to ActionText by reverting the code change — the source data is intact in `action_text_rich_texts`.

Do NOT use a `MigrationBlog` stub class (the analog used one because it was running inside an `ActiveRecord::Migration` context where the model state could be in flux; this task runs against the post-Plan-01 Blog model which is stable). Use `Blog` directly.

Do NOT require Nokogiri explicitly — it is loaded automatically as a transitive Rails dependency. If `Nokogiri` is undefined at task runtime, add `require "nokogiri"` at the top of the file as a fallback.

Add a top-of-file comment line explaining the deploy sequence: `# Deploy sequence: bin/rails db:migrate, then bundle exec rake blogs:migrate_body` (per D-01).

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `action_text_rich_texts.body` → `blogs.body` | Source data is Trix HTML written by previously-authenticated admins; flows through Nokogiri then directly into `blogs.body` via `update_column` (bypassing `sanitize_body` callback) |
| CLI invocation context | Task runs in a deployment shell, not via HTTP; operator is trusted by definition |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11 | Tampering | `update_column(:body, clean_html)` bypasses `before_save :sanitize_body` | accept (with rationale) | Source HTML originates from `action_text_rich_texts` which was itself populated by Trix output behind admin auth. Trix's output is already constrained to a small tag vocabulary that overlaps with the Phase 1 safelist (`<p>`, `<h1>`, `<strong>`, `<a>`, etc.). Running the full safelist sanitizer at migration time would (a) double the migration runtime and (b) silently drop any tag that Trix emitted but the Phase 1 safelist excludes (e.g., `<div>`) — making this an automatic content edit which is undesirable for a backfill. Operator can run `Blog.find_each { |b| b.save! }` post-migration if they want re-sanitization through the model. T-01-05 already accepted this in Plan 01. |
| T-01-12 | Information Disclosure | Progress output to stdout might include sensitive blog titles in logs | accept | Output is `Migrated N/Total posts` and `Skipping post #{blog.id}` — IDs only, no titles or content. Operator-controlled CLI; nothing reaches HTTP responses. |
| T-01-13 | Denial of Service | Loading all `ActionText::RichText` rows into memory at once for a large blog | mitigate | `Blog.find_each` iterates in batches of 1000 (Rails default). `ActionText::RichText.find_by` is a single-row lookup keyed on `(record_type, record_id, name)` — no N+1 explosion. Total memory remains bounded regardless of blog count. |
| T-01-14 | Tampering | Nokogiri stripping fails silently on malformed HTML | mitigate | `Nokogiri::HTML.fragment` is forgiving — it returns a fragment even from malformed input. `.css("action-text-attachment")` selects by tag name only (no attribute coupling) so the strip is robust against attribute variation. Idempotency check (`blog.body.present?`) prevents re-corrupting already-migrated rows on accidental re-run. |

</threat_model>
