<objective>
## Phase Goal

**As an** admin editor, **I want to** compose a blog post in the new Tiptap editor (replacing Trix) and have it persist as sanitized HTML in `blogs.body` and render on the public page, **so that** the marketing team has a working WYSIWYG editor with no XSS surface before any richer features are layered on.

This is the **walking skeleton** for Phase 1. It proves the full Stimulus ↔ esbuild ↔ Rails ↔ PostgreSQL round-trip for the new storage shape. Toolbar is intentionally minimal (Bold + H1/H2 + lists) — Plan 02 expands it. Migration of existing rows happens in Plan 03.

**Purpose:** Replace the Trix/ActionText editor cleanly. After this plan, the editor works end-to-end for NEW posts and existing posts can be opened and re-saved (their body will be `nil` until Plan 03's Rake task runs — Plan 03 handles backfill).
**Output:** Working Tiptap editor on admin blog form; new `blogs.body` column; server-side sanitization at save; public show page renders sanitized HTML; all `blog.content` callers updated; ActionText / Trix imports removed; factory updated; existing test suite green.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Rails | Admin's Tiptap editor submits arbitrary HTML in `params[:blog][:body]` — client-side sanitization is not trusted |
| Database → Rails view layer | `@blog.body` is read from DB and rendered via `raw` — depends on sanitization being applied at write time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering / Info Disclosure | `app/models/blog.rb` body persistence | mitigate | `before_save :sanitize_body` runs `Rails::Html::SafeListSanitizer.new.sanitize(body, tags: ALLOWED_TAGS, attributes: ALLOWED_ATTRIBUTES)` with `ALLOWED_TAGS = %w[p br h1 h2 h3 h4 h5 h6 ul ol li strong em a blockquote code pre]` and `ALLOWED_ATTRIBUTES = %w[href target rel]` — strips `<script>`, `<iframe>`, `<style>`, `on*` handlers, `javascript:` URLs (Rails sanitizer default behavior). Implements SEC-01. |
| T-01-02 | Tampering | `app/views/blogs/show.html.erb` public render | mitigate | View uses `<%= raw @blog.body %>` only after server-side sanitization at save time (T-01-01). View renders inside `<div class="prose prose-lg max-w-none">` so structural classes cannot be overridden by content. |
| T-01-03 | Spoofing | Tiptap output trust | mitigate | Browser-emitted HTML is never persisted as-is; the Rails controller passes `:body` straight into the model where T-01-01 runs. No `html_safe` is called on user input before sanitization. |
| T-01-04 | Information Disclosure | Sanitizer bypass via attribute injection | mitigate | `ALLOWED_ATTRIBUTES` whitelist of `href, target, rel` only. Per Rails sanitizer behavior, `href` values are filtered to safe URL schemes (`http`, `https`, `mailto`, `ftp`); `javascript:` URLs are stripped automatically — no extra filter needed. Plan 02 expanding the toolbar must NOT add `style` or `class` to `ALLOWED_ATTRIBUTES` (deferred to v2). |
| T-01-05 | Tampering | `blogs.body` direct SQL access bypassing model callbacks | accept | `update_column` and bulk `update_all` bypass the `before_save` callback. The only place this can happen in Phase 1 is the rake task in Plan 03 (which sanitizes via Nokogiri stripping of attachment nodes only — not full safelist). Accept because rake task input is ActionText DB rows, not user input. Documented for Plan 03. |
| T-01-06 | Elevation of Privilege | Admin auth on `/admin/blogs/*` | mitigate (inherited) | `Admin::BlogsController < Admin::BaseController` — `before_action :authenticate_user!` and `before_action :ensure_admin!` already enforced. No change in this plan. |
